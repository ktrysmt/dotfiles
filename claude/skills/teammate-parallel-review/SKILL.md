# 並列コードレビュー

## 起動条件
ユーザーが「並列レビュー」「観点別レビュー」「セキュリティとパフォーマンスをレビュー」「PR レビューして」等と言ったとき。複数の品質観点での並列コードレビューに使用。

## Layer 1: チーム設計

### タスク適格性チェック（必須）

- [ ] レビュー観点が2つ以上ある
- [ ] 全Teammateが Read-Only（ファイル編集不要）
- [ ] レビュー対象のファイルが特定できている（PR diff or ディレクトリ指定）

### チーム構成

```
チーム名: review-team
サイズ: Lead + N（レビュー観点数、最大4名）
トポロジー: Star型
モード: 全Teammate Read-Only（最も安全なパターン）
```

### レビュー観点の分割パターン

| 組合せ | 観点A | 観点B | 観点C（任意） |
|---|---|---|---|
| セキュリティ重視 | セキュリティ (OWASP) | パフォーマンス | テストカバレッジ |
| 品質重視 | 設計/アーキテクチャ | 可読性/保守性 | テストカバレッジ |
| インフラ変更 | セキュリティ | 可用性/耐障害性 | コスト影響 |
| API変更 | 後方互換性 | パフォーマンス | ドキュメント整合性 |

## Layer 2: ロール設計

各Reviewerのロール定義:

```
Identity: reviewer-{観点略称}
担当: {観点}の専門レビュー
Goal: {観点}の問題を発見し、重要度付きで報告する
Mode: Read-Only（ファイル編集禁止）
```

### 重要度分類（全Reviewer共通）

- Critical: 本番障害/データ損失/セキュリティ侵害の直接的リスク
- High: 性能劣化/機能不全の可能性が高い
- Medium: ベストプラクティスからの逸脱、将来的な問題リスク
- Low: コードスタイル、軽微な改善提案

### 出力フォーマット（全Reviewer共通）

```markdown
## {観点名} レビュー結果

### サマリ
- Critical: {N}件
- High: {N}件
- Medium: {N}件
- Low: {N}件

### 発見事項

#### [{重要度}] {タイトル}
- ファイル: {path}:{line}
- 問題: {何が問題か}
- 影響: {発生しうる具体的影響}
- 推奨修正: {修正方針}

### {観点名}で問題なしの領域
- {確認済みで問題なかった項目を明記}
```

## Layer 3: Spawnプロンプトテンプレート

### セキュリティレビュー用

```
あなたは reviewer-security です。{対象}のセキュリティレビューを実行してください。

## レビュー対象
{対象ファイル一覧、またはPR diffの取得コマンド}

## 差分取得（PR の場合）
git diff {base}...{head} -- {対象パス}

## レビュー観点（OWASP Top 10 準拠）
1. A01 Broken Access Control: 認可バイパスの可能性
2. A02 Cryptographic Failures: 暗号アルゴリズム、シークレット管理
3. A03 Injection: ユーザー入力の検証
4. A07 Authentication Failures: セッション管理、トークン処理
5. A08 Software Integrity: 依存パッケージの脆弱性
6. A09 Logging Failures: 機密情報のログ出力

## 受入基準
- [ ] 上記6観点すべてについてレビュー結果を記載
- [ ] 各発見にファイル:行番号の参照を含む
- [ ] 問題なしの観点も明示的に記載

## 禁止事項
- ファイルを編集しない
- パフォーマンス等、他の観点のレビューは他Reviewerの担当

完了後、SendMessage で Critical/High件数 + 最重要発見(5行以内)をLeadに返却。
```

### パフォーマンスレビュー用

```
あなたは reviewer-performance です。{対象}のパフォーマンスレビューを実行してください。

## レビュー対象
{対象ファイル一覧、またはPR diffの取得コマンド}

## レビュー観点
1. レイテンシ: ホットパスの不要な同期処理、ブロッキング呼出し
2. メモリ: メモリリークパターン、不要なオブジェクト保持
3. 同時実行: 非同期処理の適切性、競合状態
4. キャッシュ: キャッシュ戦略の妥当性、TTL設定
5. N+1: ループ内のDB/API呼出し
6. バンドルサイズ: 不要なimport、tree-shakingの阻害

## 受入基準
- [ ] 上記6観点すべてについてレビュー結果を記載
- [ ] 各発見にファイル:行番号の参照を含む
- [ ] 影響範囲の推定（リクエスト/秒、レイテンシ増加等）を記載

## 禁止事項
- ファイルを編集しない
- セキュリティ等、他の観点は他Reviewerの担当

完了後、SendMessage で Critical/High件数 + 最重要発見(5行以内)をLeadに返却。
```

### テストカバレッジレビュー用

```
あなたは reviewer-coverage です。{対象}のテストカバレッジレビューを実行してください。

## レビュー対象
{対象ファイル一覧、またはPR diffの取得コマンド}

## レビュー観点
1. 新規コードのテスト有無: 追加された関数/クラスに対応するテストが存在するか
2. エッジケース: 境界値、null/undefined、空配列、エラーパス
3. モック適切性: 外部依存のモックが適切か、過度なモックがないか
4. テスト品質: アサーションが具体的か、テスト名が意図を表現しているか
5. 統合テスト: コンポーネント間の結合をテストしているか

## 受入基準
- [ ] テストが欠落しているコードパスを全て列挙
- [ ] 各指摘にファイル:行番号の参照を含む
- [ ] 追加すべきテストケースの具体例を記載

## 禁止事項
- ファイルを編集しない
- セキュリティ/パフォーマンス観点は他Reviewerの担当

完了後、SendMessage で未カバーコードパス数 + 最重要指摘(5行以内)をLeadに返却。
```

## 実行フロー

1. ユーザーからレビュー対象と観点を受ける
2. レビュー観点を確定（ユーザー指定 or 上記パターンから選択）
3. TeamCreate でチーム作成
4. 全Reviewer を Read-Only で並列起動（Layer 3）
5. 全Reviewer 完了後、Lead が結果を統合:
   - Critical/High を優先順位付きでまとめる
   - 観点間の重複指摘を排除
   - 総合判定（Approve / Request Changes / Needs Discussion）
6. ユーザーに統合レビュー結果を報告
7. チームメイトをシャットダウン
