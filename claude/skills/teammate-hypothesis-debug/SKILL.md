# 競合仮説デバッグ

## 起動条件
ユーザーが「仮説検証」「バグの原因を並列で調べて」「competing hypothesis」「仮説を複数立てて調査」等と言ったとき。バグ報告と共に使用される。

## Layer 1: チーム設計

### タスク適格性チェック（必須）

- [ ] バグの原因候補が2つ以上ある
- [ ] 各仮説の検証が独立した対象ファイルで行える
- [ ] ファイル編集不要（調査のみ）

全てチェック: Agent Teamsを使用。
1つでも未チェック: 単一エージェントで逐次検証。

### チーム構成

```
チーム名: debug-team
サイズ: Lead + N（仮説数、最大3名）
トポロジー: Star型
モード: 全Teammate Read-Only（ファイル競合リスクゼロ）
```

### 仮説分割ルール

1. ユーザーのバグ報告から原因仮説を列挙（ユーザーが指定 or Leadが推定）
2. 各仮説に検証対象のファイル/ディレクトリを割り当て
3. 仮説間で対象範囲が重複しないよう調整
4. 重複が避けられない場合は読取のみなので許容するが、プロンプトに「他仮説の検証範囲には踏み込まない」と明記

## Layer 2: ロール設計

各Investigatorのロール定義:

```
Identity: investigator-{仮説略称}
担当: {仮説}の検証
Goal: 証拠に基づき、仮説の確認/否定/不確定を判定する
Mode: Read-Only（ファイル編集禁止）
```

### 出力フォーマット（全Investigator共通）

```markdown
## {仮説名} 検証結果

### 判定: 確認 / 否定 / 不確定

### 根拠
- [{ファイル}:{行番号}] {問題箇所の説明}

### 推奨修正（確認の場合）
- {修正箇所と修正方針}

### 追加検証が必要な場合
- {何を、どのように}
```

## Layer 3: Spawnプロンプトテンプレート

```
あなたは investigator-{仮説略称} です。デバッグチームのチームメイトとして、
以下の仮説を検証してください。

## バグ概要
{バグの症状、発生条件、発生頻度、発生開始時期}

## 検証する仮説
{仮説の具体的記述}

## 検証手順
1. {差分取得コマンド or ログ確認手順}
2. {対象ファイルの精査手順}
3. {パターン確認（リーク、競合、設定ミス等）}
4. {証拠の収集と判定}

## 対象範囲
- 精査対象: {仮説に関連するファイル/ディレクトリ}
- 参照のみ: {設定ファイル、依存定義等}

## 受入基準
- [ ] 判定（確認/否定/不確定）を根拠と共に明示
- [ ] 各根拠にファイル:行番号の参照を含む
- [ ] 確認の場合は推奨修正を記載

## 禁止事項
- ファイルを編集しない（Read-Onlyモード）
- 他の仮説（{他Investigator名}）の担当領域は調査しない
- 証拠がない推測は記載しない

完了後、SendMessage で判定 + 最重要根拠(5行以内)をLeadに返却。
```

## 実行フロー

1. ユーザーからバグ報告を受ける
2. 原因仮説を列挙（ユーザー指定 or Leadが推定、ユーザーに確認）
3. 各仮説の検証対象ファイルを特定（Layer 1）
4. TeamCreate でチーム作成
5. 各Investigator を Read-Only で並列起動（Layer 3）
6. 全Investigator 完了後、Leadが結果を突き合わせ
7. 結果パターンに応じた対応:
   - 1つ確認: その仮説に基づき修正方針を提示
   - 複数確認: 複合原因の可能性を報告
   - 全否定: 新仮説の必要性をユーザーに報告
   - 不確定あり: 追加検証手順を提示
8. チームメイトをシャットダウン
