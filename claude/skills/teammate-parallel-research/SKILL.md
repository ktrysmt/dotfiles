# 並列リサーチ

## 起動条件
ユーザーが「並列で調べて」「複数の観点から調査」「parallel research」「分担して調査」等と言ったとき。技術調査・比較調査・エビデンス収集等で使用。

注意: 本スキルは汎用リサーチ用。学術論文分析には /llm-research-paper または /evidence-review を使用すること。

## Layer 1: チーム設計

### タスク適格性チェック（必須）

- [ ] 調査軸が2つ以上ある（例: 学術/実務、技術A/技術B、国内/海外）
- [ ] 各軸の調査が独立して実行可能
- [ ] 出力が統合可能な共通フォーマットを持てる

### チーム構成

```
チーム名: research-team
サイズ: Lead + N（調査軸数、最大4名）
トポロジー: Star型
モード: Read-Only推奨（register/への書出しのみ許可）
```

### 調査軸分割パターン

| パターン | 分割方法 | 例 |
|---|---|---|
| ソース分割 | 情報源の種類で分割 | 学術論文 / 実務ブログ / コミュニティ |
| 観点分割 | 分析の観点で分割 | セキュリティ / パフォーマンス / コスト |
| 技術分割 | 比較対象で分割 | React / Vue / Svelte |
| 地域分割 | 地域・言語で分割 | 英語圏 / 日本語圏 / 中国語圏 |

### MCP分担ルール

調査軸間でMCPの重複使用を避け、コンテキスト効率を最大化する:
- Researcher-A: Scite + Arxiv（学術系）
- Researcher-B: Web Search + HackerNews + DeepWiki（実務系）
- 重複が避けられない場合: 検索クエリを明示的に分離

## Layer 2: ロール設計

各Researcherのロール定義:

```
Identity: researcher-{調査軸略称}
担当: {調査軸}に関する情報収集と整理
Goal: {調査目的}に対し、{情報源}から{件数目安}件のエビデンスを収集する
```

### 重要: 事実収集タスクでのペルソナ制限

根拠: Zheng et al. (EMNLP 2024), Basil et al. (2025)

事実収集・エビデンス収集タスクでは、ペルソナ（「あなたはAI研究の専門家です」等）を付与しない。
事実精度タスクではペルソナが改善に寄与せず、むしろ「演技」が知識想起を阻害するリスクがある。

ロール定義は「何をするか」のみに限定し、「誰であるか」の記述を避ける。

### 出力フォーマット（全Researcher共通）

register/{チーム名}-{調査軸略称}.md に書き出す:

```markdown
# {調査軸名} 調査結果

## 収集情報
| # | タイトル/名称 | 出典 | 年/日付 | 要約 | URL |
|---|---|---|---|---|---|

## 主要知見（箇条書き5-10項目）

## 情報源一覧
```

## Layer 3: Spawnプロンプトテンプレート

```
あなたは researcher-{調査軸略称} です。

## タスク
「{調査テーマ}」に関する{調査軸}の情報を収集し整理する。

## 検索戦略
以下のMCPを使用し、各クエリ最大10件を取得:

1. {MCP名} {ツール名}:
   - "{検索クエリ1}"
   - "{検索クエリ2}"
   - "{検索クエリ3}"

2. {MCP名} {ツール名}:
   - "{検索クエリ4}"

## 出力仕様
- 書き出し先: register/{チーム名}-{調査軸略称}.md
- フォーマット: 上記の共通フォーマットに従う

## 受入基準
- [ ] {件数}件以上の情報を収集
- [ ] 各情報に出典URLを記載
- [ ] 重複排除済み
- [ ] 主要知見を5-10項目に整理

## 禁止事項
- 「あなたは専門家です」等のペルソナ付与は不要（事実収集タスク）
- 情報を推測で補完しない。取得できた情報のみを記載
- 他Researcher（{他Researcher名}）の担当MCP/検索クエリを重複実行しない

完了後、SendMessage でサマリ(10行以内)+ファイルパスをLeadに返却。
```

## 実行フロー

1. ユーザーから調査テーマを受ける
2. 調査軸を決定（ユーザー指定 or Leadが提案してユーザーに確認）
3. 各軸の検索戦略とMCP分担を策定（Layer 1）
4. TeamCreate でチーム作成
5. 各Researcher を並列起動（Layer 3）
6. 全Researcher 完了後、Lead が register/ から中間ファイルを読込
7. 結果を統合し、ユーザーに報告（or results/ にレポート書出し）
8. チームメイトをシャットダウン
9. ユーザーに register/ の削除コマンドを提示
